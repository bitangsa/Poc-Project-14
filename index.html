ML Platform – Security Pattern

Secrets & Config Management for ML Jobs

A minimal, opinionated structure for keeping credentials out of notebooks and code, while still letting training and inference jobs run safely.
Scope: Single ML project
Environment: Dev / Prod
1 Project Objective
Goal driven
The core objective of this project is to keep passwords, API keys, AWS credentials (including the AWS Secret Access Key) out of notebooks and source code. Instead, all secrets live in a secure vault and are accessed at runtime using least-privilege IAM roles.
  • a. If someone sees the code or notebook, they still can’t see the underlying secrets.
  • b. If a secret is compromised, it can be rotated without touching or redeploying code.
  • c. Auditors can track which identity accessed which secret and when.
Design principle: source code is treated as non-secret; only the vault and IAM layer know the actual credentials.

4 Model & CI/CD
Quality gate
Model Security Check
The “model” aspect of this project is not a specific ML algorithm, but the static analysis model provided by AWS CodeGuru Reviewer’s secrets detector, integrated into CI/CD.
Every push or merge triggers:
  • Scanning for patterns that look like access keys, passwords, or tokens.
  • Blocking merges or surfacing warnings when secrets are found.
  • Encouraging remediation: move the value to a vault, replace code with dynamic lookup via IAM role.
Pipeline Responsibilities
Before deployment: the pipeline must assert that no new secrets are hard-coded. Only then can ML training/serving jobs be deployed.
